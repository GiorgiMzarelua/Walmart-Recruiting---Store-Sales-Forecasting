## 1. ამოცანის აღწერა

### 1.1 ამოცანა

Walmart-ის **45 მაღაზიის** დეპარტამენტების **კვირეული გაყიდვების** პროგნოზი.

- **3,331 დამოუკიდებელი დროითი მწკრივი** (`Store × Dept`)
- **Train:** 2010-02-05 → 2012-10-26 (143 კვირა, 421,570 row)
- **Test:** 2012-11-02 → 2013-07-26 (**39 კვირა**, 115,064 row)

### 1.2 მეტრიკა - WMAE

```
WMAE = Σ(wᵢ · |yᵢ − ŷᵢ|) / Σ(wᵢ),   wᵢ = 5 (თუ სადღესასწაულო კვირაა) ან 1(დანარჩენ შემთხვევაში)
```

**ეს ერთი ფორმულა მთელი პროექტის განმსაზღვრელია**:

**1. WMAE არის L1 (absolute error) მეტრიკა** → ასეთ შემთხვევაში მოდელმა **მედიანა** უნდა დააპროგნოზიროს და არა **საშუალო** (საშუალო MSE-სთვისაა ოპტიმალური). ამიტომ ყველგან L1-ზე დაფუძნებული loss ფუნქცია გამოვიყენეთ:

- **Tree მოდელები:** `objective="regression_l1"` (LightGBM) / `reg:absoluteerror` (XGBoost) / `loss_function="MAE"` (CatBoost)
- **ნეირონული ქსელები:** `loss=MAE()`

**2. სადღესასწაულო კვირები 5×-ზე იწონება** → ისინი **ტრენინგშიც** უნდა შევიდეს და არა მხოლოდ შეფასებაში. ყველა tree მოდელს გადაეცემა:
```python
sample_weight = np.where(df["IsHoliday"], 5.0, 1.0)   # make_sample_weight()
```

**3. ოთხი დღესასწაული:** Super Bowl (კვ. 6), Labor Day (36), Thanksgiving (47), Christmas (52).

**Labor Day სატესტო პერიოდში საერთოდ არ გვხვდება**, ხოლო **Super Bowl პრაქტიკულად არ გვიცვლის გაყიდვებს** (იხ. 2.4) → შედეგად, WMAE-ს ბედი წყდება **Thanksgiving-სა და Christmas-ზე**.

### 4.2 `seasonal_week_avg` — დომინანტი ფიჩერი

XGBoost-ისა და LightGBM-ის feature importance-ის მიხედვით, ეს **ფიჩერი დომინირებს ყველა დანარჩენზე**.

აგებულია იერარქიული cascade-ით (**მხოლოდ სატრენინგო მონაცემებზე** — leakage-ის თავიდან ასაცილებლად):

```
(unique_id, week) საშუალო
  ↓ არ არსებობს?
(Dept, week) საშუალო
  ↓ არ არსებობს?
(Store, week) საშუალო
  ↓ არ არსებობს?
global mean
```

არსებითად ეს არის **seasonal-naive პროგნოზი, გადაქცეული მოდელის input-ად**.

> **სწორედ ეს ხსნის, რატომ სჯობს tree მოდელები ნეირონულ ქსელებს:**
> **ხეს სეზონურობა მზა სვეტად ეძლევა. ნეირონულ ქსელს იგივე სტრუქტურა lookback window-დან უნდა ისწავლოს - შეზღუდული მონაცემებიდან.**

### 4.3 Lag ფიჩერები - გამოვცადეთ და უფრო ცუდი შედეგი მოგვცა

**პრობლემა - 39-კვირიანი ჰორიზონტი:**

სატესტო კვირა N20-ისთვის, `lag_1` ("გასული კვირის გაყიდვები") ეხება კვირას, **რომელიც ჩვენთვისაც უცნობია** - ის ჯერ უნდა დავაფრედიქთოთ. ანუ **მოკლე lag-ები ინფერენსზე ფიზიკურად არ არსებობს.**

პირველი იმპლემენტაცია (`lag_1`, `lag_2`, `lag_4`, `rolling_mean_4`, `rolling_std_4`) იძლეოდა **შესანიშნავ** ვალიდაციურ რიცხვს - მაგრამ ეს იყო **leakage**: ვალიდაციაზე `lag_1` კითხულობდა `va`-ს **ნამდვილ** გაყიდვებს (პასუხს, რომლითაც შედეგს ვზომავდით), ტესტზე კი ოთხივე სვეტი ერთსა და იმავე მნიშვნელობას იღებდა.

**გამოსავალი:** დავტოვეთ **მხოლოდ `lag_52`** (იგივე კვირა, ოღონდ შარშან) - რადგან სატესტო ფანჯრის ყოველი თარიღისთვის, 52 კვირით ადრინდელი მნიშვნელობა **ტრენინგის შიგნით ხვდება**. აგებულია როგორც **lookup ტრენინგის ისტორიიდან** (`fit_seasonal_profiles`), და **არა** `shift()`-ით პროგნოზირებად frame-ზე → იდენტურად იქცევა train/valid/test-ზე (არც leakage, არც train-serve skew).

### შედეგი - ორ დამოუკიდებელ მოდელზე რეპლიცირებული

| მოდელი | lag-ების გარეშე | `lag_52`-ით |
|---|---|---|---|
| **XGBoost** | **2,118.94** | 2,385.59 | **+266.65** |

**რატომ:** `lag_52` **ზედმეტია**. `seasonal_week_avg` იმავე ინფორმაციას **უკეთ** აწვდის - ის ასაშუალოებს რამდენიმე წელს, ნაცვლად **ერთი ხმაურიანი დაკვირვების** ნდობისა. `lag_52` მხოლოდ **ვარიაციას** უმატებს.

> **დასკვნა:** გრძელ ჰორიზონტზე (39 კვირა) lag-ები ან **გამოუთვლელია** (მოკლე lag-ები), ან **ზედმეტია** (სეზონური lag). ეს რეპლიცირებული შედეგია, არა შემთხვევითობა.

### 5.8 Deep Learning - N-HiTS

`model_experiment_nhits.ipynb` · MLflow: `NHITS_Training`

N-HiTS არის N-BEATS-ის **პირდაპირი მემკვიდრე** - იმავე ავტორების მიერ შექმნილი, გაუმჯობესებული ვერსია. მისი მთავარი სიახლე არის **multi-rate hierarchical pooling**: მოდელი შემავალ სიგნალს **რამდენიმე სხვადასხვა სიხშირეზე** ამუშავებს, რაც სპეციალურად **გრძელი ჰორიზონტისთვის** არის გათვლილი.

ჩვენი ჰორიზონტი (39 კვირა) სწორედ ის შემთხვევაა, რომლისთვისაც N-HiTS შეიქმნა → **გაუმჯობესებას ველოდით.**

| კონფიგურაცია | WMAE |
|---|---|
| default pooling `[2,2,1]` | 2,797.45 |
| **pooling გამორთული `[1,1,1]`** | **2,651.03** |
| *(N-BEATS-ის საუკეთესო)* | *2,588.18* |

**საწინააღმდეგო შედეგი:** N-HiTS **ჩამორჩა** N-BEATS-ს. და როცა მისი **მთავარი ინოვაცია (pooling) გამოვრთეთ** - შედეგი **გაუმჯობესდა** (2,797 → 2,651) და მიუახლოვდა N-BEATS-ს.

**ახსნა:** N-HiTS-ის pooling შექმნილია იმისთვის, რომ სიგნალი **სხვადასხვა სიხშირეზე** დაშალოს — ეს კარგად მუშაობს მაშინ, როცა მონაცემებში **მრავალი სხვადასხვა ციკლია** ერთმანეთში გადახლართული (მაგ. საათობრივი ელექტროენერგია: დღიური, კვირეული და სეზონური რიტმები ერთდროულად, ასობით ნაბიჯზე).

ჩვენს მონაცემებში კი ასეთი სირთულე **უბრალოდ არ არის**:

> **დასკვნა: ახალი არ ნიშნავს აპრიორი უკეთესს.** ჩვენ არ დავეყრდენით რეპუტაციას - გავტესტეთ და აღმოვაჩინეთ, რომ არქიტექტურის **საკვანძო ინოვაცია საზიანოა** ჩვენი მონაცემების სტრუქტურისთვის.

#### Forecast vs Actual — რატომ აგებენ ნეირონული ქსელები

ის გრაფიკი, რაც უშუალოდ ნოუთბუქშია **ვიზუალურად ხსნის მთელი deep learning სექციის შედეგს**: **პროგნოზი თითქმის ბრტყელია**, მაშინ როცა რეალობა მკვეთრად მერყეობს და დიდი პიკები აქვს (განსაკუთრებით Christmas-ის წინ).

მოდელი იჭერს **საშუალო დონეს**, მაგრამ **არა ვარიაციას** - ის საშუალოსკენ იხრება (**mean reversion**).

**და ეს არ არის შეცდომა - ეს ოპტიმალური ქცევაა.** WMAE არის **L1** მეტრიკა. როცა მოდელს **არ შეუძლია იწინასწარმეტყველოს, *როდის* მოხდება პიკი**, საშუალოსკენ გადახრა **მინიმუმამდე ამცირებს** mean absolute error-ს.

**რეგისტრირებულია:** `walmart_nhits`

### 5.9 Deep Learning - DLinear

`model_experiment_dlinear.ipynb` · MLflow: `DLinear_Training`

| კონფიგურაცია | WMAE | ქცევა |
|---|---|---|
| robust scaler, lr=1e-4 | 3,490.46 | ბრტყელი |
| **robust scaler, lr=1e-2** | **3,294.91** | ბრტყელი |
| standard scaler, input=104 | **5,481.36** | ხტუნავს, აცილებს |
| + signed_log1p target | 3,409.37 | ბრტყელი |

**სამი ღირებული აღმოჩენა:**

**1. Bias-variance tradeoff თვალსაჩინოდ.** `standard` scaler-ით პროგნოზი აღარ იყო უცვლელი - ის **ცდილობდა** პიკების დაჭერას. მაგრამ WMAE **გაუარესდა 66%-ით** (3,294 → 5,481), რადგან პიკებს **ხან ძალიან მაღლა ისროდა ხან პირიქით**.
> **როცა მოდელს პიკის *დროის* განსაზღვრა არ შეუძლია, საშუალოსთან ახლოს დარჩენა (high bias) სჯობს პიკებზე "ფსონის დადებას" (high variance)** — არასწორ დროს დაჭერილი პიკი **ორმაგ** შეცდომას იძლევა:

**2. სხვადასხვა არქიტექტურას სხვადასხვა learning rate სჭირდება.** N-BEATS-მა **დაბალი** lr (1e-4) მოისურვა, DLinear-მა — **მაღალი** (1e-2). პატარა წრფივი მოდელები მაღალ lr-ს ითხოვენ.

**3. log ტრანსფორმაცია **არ** დაეხმარა** (3,294 → 3,409). ჰიპოთეზა, რომ ნეირონული ქსელების ჩამორჩენა skew-ით იყო გამოწვეული — **გავტესტეთ და უარვყავით**.

**რეგისტრირებულია:** `walmart_dlinear`

## 8. აღმოჩენილი შეცდომები და Data Leakage

ეს სექცია განზრახ დეტალურია. ჩვენ აღმოვაჩინეთ და გავასწორეთ **შვიდი სერიოზული ბაგი**, რომელთაგან **ხუთი data leakage-ს ან train-serve skew-ს იწვევდა**. თითოეული მათგანი **ლამაზ, მაგრამ ყალბ WMAE-ს** აჩვენებდა.

---

### 8.1 XGBoost: სრულ `train`-ზე ტრენინგი, `va`-ზე შეფასება

```python
bundle = fit_xgboost(train, ...)      # ტრენინგი სრულ train-ზე
preds  = predict_xgboost(bundle, va)  # შეფასება va-ზე... მაგრამ va ⊆ train !
```

`va` არის `train`-ის **ქვესიმრავლე** → მოდელმა **ტრენინგისას იხილა ვალიდაციის პასუხები**. ორმაგი leakage: **მოდელმაც** და **`fit_seasonal_profiles(train)`-მაც**.

| | |
|---|---|
| **ყალბი WMAE** | **958.36** |
| **გასწორება** | `fit_xgboost(tr, ...)` |

---

### 8.2 Lag ფიჩერების row-alignment ბაგი

`add_lag_features` ასორტირებდა `df.sort_values(["unique_id", "Date"])`-ით. მაგრამ `unique_id` არის **სტრიქონი** → სორტირება **ლექსიკოგრაფიულია**:

```
სტრიქონული:  "1_1", "1_10", "1_11", ..., "1_2", ..., "10_1", ..., "2_1"
რიცხვითი:    "1_1", "1_2",  ...,      "1_10", ..., "2_1",  ..., "10_1"
```

ფუნქცია აბრუნებდა **დასორტირებულ** frame-ს, ხოლო `wmae()` ადარებდა **თავდაპირველ** `va`-ს **პოზიციურად** → **თითოეული პროგნოზი სხვა მწკრივის რეალურ გაყიდვას ედარებოდა**.

| | |
|---|---|
| **ყალბი WMAE** | **19,409.78** (≈ საშუალო გაყიდვების დონე - misalignment-ის კლასიკური ხელწერა) |
| **გასწორება** | სორტირება **მთლიანად ამოვიღეთ** - `lag_52` ახლა lookup-ია, არა `shift()` |

---

### 8.3 მოკლე lag-ები კითხულობდნენ ვალიდაციის პასუხებს

`lag_1`, `lag_2`, `lag_4`, `rolling_mean_4` გამოითვლებოდა **იმ frame-იდან, რომელსაც ვპროგნოზირებდით**:

| | ვალიდაციაზე (`va`) | ტესტზე (`test`) |
|---|---|---|
| `lag_1` | **ნამდვილი** გასული კვირის გაყიდვები ← **leakage!** | NaN → `seasonal_week_avg` |
| `lag_2`, `lag_4` | ნამდვილი მნიშვნელობები | NaN → `seasonal_week_avg` |
| `rolling_mean_4` | ნამდვილი მნიშვნელობები | NaN → `seasonal_week_avg` |
| `rolling_std_4` | ნამდვილი | → **0** |

ანუ ტესტზე **ოთხი იდენტური სვეტი** და ერთი კონსტანტა. **სრული train-serve skew.**

| | |
|---|---|
| **ყალბი WMAE** | **679 / 2,081** |
| **გასწორება** | მოკლე lag-ები **მთლიანად ამოვიღეთ** |

---

### 8.4 `lag_52` = 100% NaN ვალიდაციასა და ტესტზე

`g.shift(52)` გამოიყენებოდა **გადაცემულ frame-ზე**. მაგრამ `va` და `test` თითო მწკრივზე მხოლოდ **39 row**-ს შეიცავს → **52-ით shift 39-row-იან ჯგუფში შეუძლებელია** → **სულ NaN**. და `fillna` ბლოკი `lag_52`-ს **არასოდეს ავსებდა**.

**შედეგი:** მოდელი ტრენინგდებოდა **რეალურ** `lag_52`-ზე და ინფერენსზე იღებდა **სულ NaN-ს**.

**გასწორება:** `lag_52` აიგო როგორც **lookup ტრენინგის ისტორიიდან**:

```python
t["target_date"] = t["Date"] + pd.Timedelta(weeks=52)   # თარიღი, რომელსაც ეს მნიშვნელობა პროგნოზირებს
lag52_series = t.groupby(["unique_id", "target_date"])["Weekly_Sales"].mean()
```

ახლა იგი **იდენტურად იქცევა** train / valid / test-ზე → **არც leakage, არც skew**.

### 8.7 `seasonal_week_avg`-ის self-leak ტრენინგზე

`tr` მოიცავს ~90 კვირას → `(unique_id, week)` უჯრედების **~27%** შეიცავს **მხოლოდ ერთ დაკვირვებას** → `mean()` აბრუნებს **თავად ამ row-ის გაყიდვას**:

```
TRAIN (uid,week) cells with only ONE observation: 26.9%
  corr(seasonal_week_avg, y) on those rows: 1.0000   ← ეს არის სამიზნე!
```

ეს **არ ზრდის** ვალიდაციის ქულას (ვალიდაციაზე ფიჩერი პატიოსანია), მაგრამ **მოდელს არასწორად აკალიბრებს** — ის სწავლობს, რომ "y = seasonal_week_avg" **სრულყოფილი** წესია, ხოლო ინფერენსზე იღებს **ხმაურიან შეფასებას**.

**რეკომენდებული გასწორება (მომავალი სამუშაო):** leave-one-out encoding `(sum − own_y) / (cnt − 1)`.

### 9.2 ბრტყელი პროგნოზი არ არის შეცდომა - ის **ოპტიმალური სტრატეგიაა**

Forecast-vs-actual გრაფიკებმა აჩვენა, რომ ყველა უნივარიატული ნეირონული ქსელი **საშუალოსკენ იხრება** (mean reversion) - პროგნოზი თითქმის ბრტყელია.

**ეს არ არის მოდელის უუნარობა.** WMAE არის **L1** მეტრიკა. როცა მოდელს არ შეუძლია **პიკის დროის** განსაზღვრა, საშუალოსკენ ხრა **მინიმიზებს** საშუალო აბსოლუტურ შეცდომას.

**🔬 ექსპერიმენტული დადასტურება (DLinear):** სკალერის შეცვლით პროგნოზები "ცოცხალი" გავხადეთ (ხტოდნენ, პიკებს მისდევდნენ) → WMAE **გაუარესდა 66%-ით** (3,294 → 5,481).
